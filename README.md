### 33 규칙에 대한 부연 설명

33 규칙은 흑돌과 백돌의 공정성을 위해 존재하는 규칙입니다. 

오목을 진행하면 돌을 두는 순서는 무조건 흑돌이 먼저입니다.

오목은 5줄로 연속된 돌을 먼저 만드는 쪽이 승리하는 게임이지만, 양쪽 방향이 막히지 않은 3연속의 돌이 있는 상태로 본인의 차례가 되면 이기는 것과 마찬가지 입니다.

그리고 이 3줄이 돌 1개를 추가했을 때 2개가 되는 경우가 있습니다.

<img src="33 직전.png" width="400"> 
안쪽이 채워진 흑돌이 (6, 5)좌표에 돌을 두면 3줄이 가로로 1개 세로로 1개 총 2개가 만들어 집니다.

이를 33 이라고 부르는 것이고 이런 상황이 온다면 백돌은 어떤 수를 두더라도 패배할 수 밖에 없는 것입니다.

그런데 33은 항상 먼저 수를 두는 흑돌이 만들기가 훨씬 유리합니다. 

<img src="33 막힌 사진.png" width="400"> 
그렇기에 흑돌이 33을 만드는 것을 금지하는 규칙이 존재하는 것입니다.

그러나 저런 형태가 무조건 안되는 것은 아닙니다.

앞에서 말씀드렸던 것처럼 33에서 금지되는 것은 돌 1개를 추가했을 때 없던 3줄이 동시에 2개가 추가되는 상황입니다.

<img src="33아닌 사진.png" width="400"> 
그렇기 때문에 이미 세로 줄을 만들어 놓은 상태로 (8, 5)좌표를 만드는 것은 백돌 입장에서는 이미 세로3줄을 막을 상황이 있었으나 막지 않은 것이고 (8,5) 좌표의 돌로 생성된 3줄은 가로3줄 1개 뿐이기 때문에 취소되지 않고 게임이 그대로 진행되는 모습입니다.

제공된 사진에서는 빠르게 33을 만들기 위해 백돌을 무작위로 좌측상단에 몰아놓았지만 실제 게임에서는 백돌이 승리를 포기하지 않는 이상 말이되지 않는 상황이기에 이 부분은 공정성과는 관련이 없다고 판단해 막히지 않는 것입니다.

### 아쉬운 점
- 구현된 오목을 플레이하려면 반드시 2명 이상의 사람이 요구됩니다. 개인적인 욕심으로 혼자 인공지능과 상대하는 기능을 구현해 보고 싶었지만 아무곳에나 돌을 랜덤하게 놓는 기능이 아닌 이기려는 목적을 가지고 더 승률이 높은 곳을 계산해 돌을 놓는 기능은 오목 실력의 부족함과 코드의 구현력 부족으로 실현되지 못해 아쉽다고 생각합니다.
- 한번도 협업을 진행해 본적이 없는 상태에서 진행된 과제인 만큼 처음 느껴보는 불편함이 몇 가지 있었습니다. 개개인이 기능을 구현할 때 생각하는 방식이나 코드를 구현하는 방식이 다른데 이를 그냥 작성하고 넘기니 확인하고 다음 작업을 수행할 때 팀원이 어떤 생각과 의도를 가지고 코드를 작성하거나 수정했는지 파악하는 것이 꽤 번거롭다고 느껴졌습니다. 그래서 프로젝트의 중간부터는 Pull Request의 설명에도 어떤 코드를 몇 쨰줄에 추가했는지나 코드의 몇몇 내용을 주석에 추가하는 방식으로 불편함을 덜어내려고 했으나 초반에 그런 불편함을 예상하지 못하고 진행된 것이 아쉬웠습니다.
